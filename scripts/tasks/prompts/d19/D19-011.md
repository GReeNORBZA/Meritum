# Task D19-011: WCB claim export function — full WCB claim details + child records CSV generator

## What to Build

Create a WCB claim CSV export function. This should be added to a new file `apps/api/src/domains/platform/exports/wcb-export.service.ts` or an existing export service location.

**Function signature:**

```typescript
export async function exportWcbClaimsCsv(
  deps: WcbExportDeps,
  physicianId: string,
): Promise<string>
```

**Dependency interface:**

```typescript
export interface WcbExportDeps {
  db: DrizzleDb; // Drizzle database instance for direct queries
}
```

**Implementation:**

Query all WCB claim data for the given physician. The WCB domain schema includes these tables that need to be joined and exported:

1. **WCB Claims** (main record): `wcb_claim_details` — claim_id, wcb_claim_number, employer_name, injury_date, form_type, status, created_at, etc.
2. **Injuries**: `wcb_injuries` — injury descriptions, body parts, ICD codes linked to each claim.
3. **Prescriptions**: `wcb_prescriptions` — medication details linked to each claim.
4. **Consultations**: `wcb_consultations` — specialist consultation records linked to each claim.
5. **Work Restrictions**: `wcb_work_restrictions` — restriction type, duration, linked to each claim.
6. **Invoice Lines**: `wcb_invoice_lines` — billing line items for each claim.

**CSV Format:**

Generate a denormalized CSV where each row represents a claim with child records flattened. Use the following column structure:

```
claim_id,wcb_claim_number,employer_name,injury_date,form_type,status,
injuries(JSON array),prescriptions(JSON array),consultations(JSON array),
work_restrictions(JSON array),invoice_lines(JSON array),
created_at,updated_at
```

For child records, serialize as JSON arrays within the CSV cell (escaped properly). This ensures all data is captured without losing the one-to-many relationships.

Alternative approach: generate one main claims CSV and separate CSVs for each child entity, all referencing claim_id. Choose whichever approach is simpler, but document the format in a header comment.

**Critical: Physician scoping.**

Every query MUST include `WHERE provider_id = :physicianId` (or the equivalent join through claims -> provider_id). This ensures no cross-tenant data leakage.

**CSV generation:**

Use a CSV generation approach consistent with existing export functions in the codebase. If the codebase already uses a CSV library (like `csv-stringify`, `papaparse`, or manual string building), follow that pattern. If no CSV library exists, use simple string concatenation with proper escaping:

```typescript
function escapeCsvField(value: string | null): string {
  if (value === null || value === undefined) return '';
  const str = String(value);
  if (str.includes(',') || str.includes('"') || str.includes('\n')) {
    return `"${str.replace(/"/g, '""')}"`;
  }
  return str;
}
```

## Critical Security Rules

- ALL queries MUST be scoped by `physicianId`. This is the tenant isolation boundary.
- Never query WCB data without the physician filter.
- Do not include any data from other physicians, even in error cases.
- The export function must work within the context of the physician's own data only.
