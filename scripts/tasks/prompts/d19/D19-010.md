# Task D19-010: Service: generateFullPortabilityExport(physicianId) — ZIP bundle with 6 CSV data types

## What to Build

Create the data portability export orchestration function. This can live in a new file `apps/api/src/domains/platform/data-portability.service.ts` or be added to the existing platform service.

**Function signature:**

```typescript
export async function generateFullPortabilityExport(
  deps: DataPortabilityDeps,
  physicianId: string,
): Promise<PortabilityExportResult>
```

**Dependency interface:**

```typescript
export interface DataPortabilityDeps {
  // Existing export functions (reuse)
  claimExport: {
    exportAhcipClaimsCsv(physicianId: string): Promise<string>; // CSV string

  // New export functions (built in D19-011 and D19-012)
```

**Return type:**

```typescript
export interface PortabilityExportResult {
  zipBuffer: Buffer;
  filename: string;
  generatedAt: Date;
  contents: string[]; // List of filenames in the ZIP
}
```

**Implementation logic (per spec B4-3):**

1. Validate the physician has access to data export:
   - Active subscription: always allowed (Feature `DATA_EXPORT` is available).
   - Cancelled subscription: allowed only within the 30-day deletion grace period (`deletionScheduledAt > now()`).
   - Otherwise: throw `BusinessRuleError('Data export is not available')`.

2. Generate all 6 CSVs in parallel using `Promise.all`:
   ```
   1. ahcip_claims.csv     — from claimExport.exportAhcipClaimsCsv(physicianId)
   2. wcb_claims.csv        — from wcbExport.exportWcbClaimsCsv(physicianId)
   3. patients.csv           — from claimExport.exportPatientsCsv(physicianId)
   4. assessments.csv        — from assessmentExport.exportAssessmentsCsv(physicianId)
   5. analytics_summary.csv  — from analyticsExport.exportAnalyticsCsv(physicianId)
   6. intelligence_history.csv — from intelligenceExport.exportIntelligenceCsv(physicianId)
   ```

3. Bundle into a ZIP using the `archiver` npm package (or `yazl` or Node.js built-in `zlib` with tar):
   ```typescript
   import archiver from 'archiver';
   // or use a simpler approach with JSZip or similar

   const zip = archiver('zip', { zlib: { level: 9 } });
   // Add each CSV as a file entry
   zip.append(ahcipCsv, { name: 'ahcip_claims.csv' });
   zip.append(wcbCsv, { name: 'wcb_claims.csv' });
   zip.append(patientsCsv, { name: 'patients.csv' });
   zip.append(assessmentsCsv, { name: 'assessments.csv' });
   zip.append(analyticsCsv, { name: 'analytics_summary.csv' });
   zip.append(intelligenceCsv, { name: 'intelligence_history.csv' });
   ```

4. Generate filename: `meritum_export_${physicianId}_${YYYY-MM-DD}.zip`

5. Audit log with action `'data_export.full_portability'` including the list of files generated.

6. Return the ZIP buffer, filename, generation timestamp, and list of contents.

**Important:** The actual CSV generation functions for WCB claims, assessments, analytics, and intelligence are implemented in D19-011 and D19-012. For now, define the interfaces and orchestration. If those dependencies don't exist yet, the tests should mock them.

## Critical Security Rules

- All export functions MUST be physician-scoped. The `physicianId` parameter must be used to filter data at the repository layer — no cross-tenant data leakage.
- The ZIP is generated server-side and either streamed to the client or stored temporarily. Never expose raw file paths.
- Audit log every export request.
- Verify the physician's subscription status before allowing export.
