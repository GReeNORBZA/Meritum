# Task D19-012: Assessment + analytics + intelligence export CSV generators for portability bundle

## What to Build

Create three CSV export functions for the data portability bundle. These can be placed in:
- `apps/api/src/domains/platform/exports/assessment-export.service.ts`
- `apps/api/src/domains/platform/exports/analytics-export.service.ts`
- `apps/api/src/domains/platform/exports/intelligence-export.service.ts`

Or combined into a single file if preferred.

### 1. Assessment Export

**Function signature:**

```typescript
export async function exportAssessmentsCsv(
  deps: { db: DrizzleDb },
  physicianId: string,
): Promise<string>
```

**Data to include (from existing tables):**

Query the following data scoped to the physician:

- **AHCIP assessment results**: From assessment-related tables (e.g., `ahcip_claim_details` assessment fields, or dedicated assessment tables if they exist). Include:
  - `claim_id`, `assessment_date`, `assessment_type`, `result`, `score`, `notes`
- **Rejection details**: From claim rejection/response records. Include:
  - `claim_id`, `rejection_code`, `rejection_reason`, `rejection_date`, `response_action`
- **Batch outcomes**: From batch submission records. Include:
  - `batch_id`, `submission_date`, `total_claims`, `accepted_count`, `rejected_count`, `total_amount`, `status`

**CSV columns:**

```
record_type,claim_id,batch_id,date,type,result_or_code,description,amount,status
```

Use `record_type` to distinguish rows: `ASSESSMENT`, `REJECTION`, `BATCH_OUTCOME`.

### 2. Analytics Export

**Function signature:**

```typescript
export async function exportAnalyticsCsv(
  deps: { db: DrizzleDb },
  physicianId: string,
): Promise<string>
```

**Data to include:**

Query from `analytics_cache` or equivalent analytics/dashboard tables:

- Dashboard summary data for the physician:
  - `metric_name`, `metric_value`, `period_start`, `period_end`, `calculated_at`
- Generated report history from `generated_reports`:
  - `report_id`, `report_type`, `generated_at`, `parameters`, `status`

**CSV columns:**

```
record_type,id,metric_or_report_name,value,period_start,period_end,generated_at,parameters,status
```

Use `record_type` to distinguish: `METRIC`, `REPORT`.

### 3. Intelligence Export

**Function signature:**

```typescript
export async function exportIntelligenceCsv(
  deps: { db: DrizzleDb },
  physicianId: string,
): Promise<string>
```

**Data to include:**

Query from `ai_suggestion_events` or equivalent intelligence tables:

- Suggestion history for the physician:
  - `event_id`, `suggestion_type`, `suggestion_text`, `claim_id`, `confidence_score`, `user_action` (accepted/dismissed), `action_at`, `created_at`

**CSV columns:**

```
event_id,suggestion_type,suggestion_text,claim_id,confidence_score,user_action,action_at,created_at
```

### General rules for all three functions:

- **Physician scoping**: Every query MUST filter by `physicianId` / `provider_id`. No cross-tenant data.
- **Empty data**: Return CSV with header row only if no data exists.
- **CSV escaping**: Use the same CSV field escaping approach as D19-011.
- **Date formatting**: Use ISO 8601 format for all dates.
- **Null handling**: Represent null values as empty strings in CSV.

```typescript
  it('includes METRIC records from analytics cache');
  it('includes REPORT records from generated reports');
  it('returns header-only CSV when no analytics data exists');
  it('scopes all data to the given physicianId');
});

describe('exportIntelligenceCsv', () => {
  it('returns CSV with header row');
  it('includes suggestion events with action status');
  it('includes both accepted and dismissed suggestions');
  it('returns header-only CSV when no intelligence data exists');
  it('scopes all data to the given physicianId');
  it('does not include suggestions from other physicians');
});
```

## Critical Security Rules

- ALL queries MUST be scoped by `physicianId`. This is non-negotiable.
- AI suggestion text may contain references to patient data. The export is for the physician's own use, so this is acceptable. But ensure no data from other physicians leaks.
- Analytics data is physician-scoped via the existing analytics cache architecture.
- Never include raw model weights or system prompts in the intelligence export.
