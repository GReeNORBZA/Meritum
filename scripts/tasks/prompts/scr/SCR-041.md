# Task SCR-041: Tighten GR 4.4.8 referral parsing to scoped section only

## What to Build

Replace the `parseGR448` function in `scripts/enrich-hsc-data.ts` with a version that only extracts HSC codes from the GR 4.4.8 subsection, not the entire GR 4 page.

### Current Problem

The function has three "strategies" that progressively widen scope:
- Strategy 1: Links with `/fee-navigator/hsc/` hrefs (all links on the page, not scoped to 4.4.8)
- Strategy 2: Text near "4.4.8" and "referring" in the full text
- Strategy 3: Every `<li>`, `<p>`, `<td>` on the page where element text contains "referr" or "4.4.8"

Strategy 3 is especially problematic: it finds HSC codes from GR 4.1, 4.2, 4.3, etc., that happen to be in elements mentioning "referral" in a different context. This creates false positive referral requirements.

### Solution

Replace with a single scoped extraction:

```typescript
function parseGR448(html: string): {
  requiresReferral: Set<string>;
  selfReferralBlocked: Set<string>;
} {
  const $ = cheerio.load(html);
  const requiresReferral = new Set<string>();
  const selfReferralBlocked = new Set<string>();

  const fullText = $('body').text();

  // Step 1: Isolate the 4.4.8 section text only
  // Bounded by: start of "4.4.8" → start of next sub-section (4.4.9, 4.5, 4.6, or end)
  const section448Match = fullText.match(
    /4\.4\.8\b([\s\S]*?)(?=\b4\.4\.9\b|\b4\.5\b|\b4\.6\b|\b4\.7\b|\b5\.\b|$)/i,
  );

  if (!section448Match) {
    console.warn('  [WARN] Could not locate GR 4.4.8 section in GR 4 page');
    return { requiresReferral, selfReferralBlocked };
  }

  const sectionText = section448Match[0];
  console.log(`  GR 4.4.8 section: ${sectionText.length} chars extracted from GR 4 page`);

  // Step 2: Extract HSC codes from the scoped section text
  // Pattern: XX.XXY or XX.XXYY (standard Alberta HSC format) with optional asterisk
  const codePattern = /\b(\d{2}\.\d{2,3}[A-Z]{0,3})\s*(\*)?/g;
  let match;
  while ((match = codePattern.exec(sectionText)) !== null) {
    requiresReferral.add(match[1]);
    if (match[2] === '*') {
      selfReferralBlocked.add(match[1]);
    }
  }

  // Step 3: Also extract from links within the GR 4 page, but ONLY if the link's
  // surrounding context contains "4.4.8" (not just "referral" which is too broad)
  $('a[href*="/fee-navigator/hsc/"]').each((_i, el) => {
    const href = $(el).attr('href') ?? '';
    const codeMatch = href.match(/\/fee-navigator\/hsc\/([^?&#]+)/);
    if (!codeMatch) return;

    // Check if the nearest parent section mentions 4.4.8 specifically
    const parentText = $(el).closest('div, section, li, td, tr').text();
    if (parentText.includes('4.4.8')) {
      const code = decodeURIComponent(codeMatch[1]);
      requiresReferral.add(code);
    }
  });

  console.log(
    `  GR 4.4.8 parsed: ${requiresReferral.size} codes require referral, ` +
    `${selfReferralBlocked.size} self-referral blocked`
  );

  return { requiresReferral, selfReferralBlocked };
}
```

**Key changes:**
1. Section isolation uses regex to extract only the 4.4.8 subsection text from the GR 4 page.
2. Code extraction runs only within the scoped section.
3. Link-based extraction checks for "4.4.8" context (not just "referral").
4. The three overlapping strategies are replaced by a single coherent approach.
5. Added logging for audit trail.
