# Task D18-042: Security: input validation + business rules enforcement

## What to Build

Create `apps/api/test/security/platform/referral.input.security.ts`.

Test that all referral business rules are enforced and that input validation prevents abuse.

### Same-Practice Referral Blocking

Per Spec B3-2: "Validates referrer and referred are NOT on the same practice."

**Setup:** Create a practice with Physician A (referrer) and Physician B (referred), both as active members.

- Physician B attempts to redeem Physician A's referral code → must be rejected
- Error: `SAME_PRACTICE_REFERRAL`
- This is because adding a physician to a practice is a seat addition, not a referral

**Edge cases:**
- Physician A is active on Practice X, Physician B was removed from Practice X (is_active = false) → referral SHOULD be allowed (they're no longer on the same practice)
- Physician A is on Practice X, Physician C is on Practice Y → referral SHOULD be allowed (different practices)
- Neither physician is on any practice → referral SHOULD be allowed

### 3-Per-Year Cap Enforcement

Per Spec B3-2: "Checks 3-per-year cap: count QUALIFIED or CREDITED redemptions in the referrer's current anniversary year. If >= 3, transition to EXPIRED instead."

**Setup:** Create a referrer with 3 already-CREDITED redemptions in anniversary year 1.

- A 4th referral qualifies → must transition to EXPIRED, not QUALIFIED
- Cap counts QUALIFIED + CREDITED only (not PENDING, not EXPIRED)
- Cap is per anniversary year, not calendar year
- A referral in anniversary year 2 should succeed even if year 1 had 3 credits

### Re-Referral of Existing Subscriber Blocked

Per Spec B3-2: "Validates referred user has NEVER had a Meritum subscription (new signups only)."

- User with active subscription attempts to be referred → rejected (`REFERRED_NOT_NEW`)
- User with cancelled subscription attempts to be referred → rejected (`REFERRED_NOT_NEW`)
- User who has never had any subscription → allowed

### Input Validation Tests

**Referral code format:**
- Code shorter than 8 chars → 400
- Code longer than 8 chars → 400
- Code with lowercase letters → 400
- Code with special characters → 400
- Empty code → 400
- SQL injection attempt in code field (`'; DROP TABLE--`) → 400 (rejected by regex)
- XSS attempt in code field (`<script>alert(1)</script>`) → 400

**Redemption ID format:**
- Non-UUID redemption ID → 400
- SQL injection in redemption ID → 400

**Credit target validation:**
- Invalid target value (not PRACTICE_INVOICE or INDIVIDUAL_BANK) → 400
- Empty target → 400
- SQL injection in target field → 400

## Critical Security Rules

- All business rule validations must be enforced at the **service layer**, not just the route handler. Even if someone bypasses the route validation, the service must reject invalid operations.
- The same-practice check requires querying `practice_memberships` (from Batch 1). If this table doesn't exist yet, the test should verify the code path exists and is called, even if the table is empty.
- The 3-per-year cap is a hard business rule with financial implications. It must never be bypassed, even under race conditions. Consider using a database transaction with row-level locking when checking the count and creating the qualification.
- Input validation (Zod schemas) provides the first line of defense; service-layer checks provide the second.
