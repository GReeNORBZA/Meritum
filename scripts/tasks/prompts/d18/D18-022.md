# Task D18-022: Service: checkReferralQualification scheduled job

## What to Build

Add the `checkReferralQualification` function to the referral service. This is a scheduled job that runs periodically (e.g., daily) to transition PENDING referrals to QUALIFIED or EXPIRED.

**Function signature:**
```typescript
checkReferralQualification(): Promise<{ qualified: number; expired: number; skipped: number }>
```

**Logic (Pricing Gap Closure Spec B3-2):**

1. **Find all PENDING redemptions:**
   - Call `findPendingRedemptions()`

2. **For each PENDING redemption, check if the referred user has completed one full billing cycle:**
   - Look up the referred user's subscription
   - Check if at least one `payment_history` record exists with `status = 'PAID'` for that subscription (this means the referred user has made at least one payment, completing one billing cycle)
   - The `qualifying_event_at` should be set to the `paid_at` date of that first payment
   - If no paid payment exists yet, skip this redemption (remains PENDING)

3. **Calculate credit value** (1 month at referrer's current rate):
   - Look up the referrer's current subscription `plan`
   - Calculate the monthly equivalent:
     - `EARLY_BIRD_MONTHLY` → `199.00`
     - `STANDARD_MONTHLY` → `279.00`
     - `STANDARD_ANNUAL` → `265.08` ($3,181 / 12, rounded to 2 decimal places)
     - `CLINIC_MONTHLY` → `251.10`
     - `CLINIC_ANNUAL` → `238.58` ($2,863 / 12, rounded to 2 decimal places)
   - Use `SubscriptionPlanPricing` from the constants to derive these values dynamically where possible
   - If referrer has no active subscription (e.g., cancelled since referring), transition to `EXPIRED` instead

4. **Check the 3-per-year cap:**
   - Call `countQualifiedOrCreditedByReferrerAndYear(referrerUserId, anniversaryYear)`
   - If count >= `REFERRAL_MAX_CREDITS_PER_YEAR` (3): transition to `EXPIRED`
   - Otherwise: proceed to qualify

5. **Transition based on referrer's billing situation:**

   **If referrer is on a clinic tier** (plan is `CLINIC_MONTHLY` or `CLINIC_ANNUAL`):
   - Transition to `QUALIFIED`
   - Set `credit_month_value_cad` to the calculated value
   - Set `qualifying_event_at`
   - **Do NOT auto-apply.** The clinic physician must choose between PRACTICE_INVOICE and INDIVIDUAL_BANK (handled by D18-023).
   - Emit a notification to the referrer about the credit choice (if notification system exists, otherwise add a TODO comment)

   **If referrer is on an individual plan** (STANDARD_MONTHLY, STANDARD_ANNUAL, EARLY_BIRD_MONTHLY):
   - Transition to `QUALIFIED`
   - Set `credit_month_value_cad` and `qualifying_event_at`
   - Then immediately call `applyReferralCredit(redemptionId)` to auto-apply the credit to their next invoice
   - This means individual-plan referrers get auto-applied credits with no choice needed

6. **Return summary** of processed redemptions.

**Scheduling:** Register this function as a scheduled job (cron or equivalent). Follow the existing pattern for scheduled jobs in the codebase (look at `checkEarlyBirdExpiry` from Batch 2 or dunning jobs from D12-022 for the pattern).

```typescript
describe('checkReferralQualification', () => {
  it('transitions PENDING to QUALIFIED when referred user has paid');
  it('skips PENDING redemptions where referred user has no payment yet');
  it('calculates credit value as 1 month at referrer early bird rate ($199.00)');
  it('calculates credit value as 1 month at referrer standard monthly rate ($279.00)');
  it('calculates credit value as 1 month at referrer standard annual rate ($265.08)');

  it('calculates credit value as 1 month at referrer clinic annual rate ($238.58)');
  it('transitions to EXPIRED when 3-per-year cap is reached');
  it('counts only QUALIFIED and CREDITED for the cap, not PENDING or EXPIRED');
  it('caps per anniversary year, not calendar year');

  it('does NOT auto-apply credit for clinic tier referrers (waits for choice)');

  it('returns correct summary counts');

```

## Critical Security Rules

- Credit value calculation must use `decimal` arithmetic, never floating point.
- The 3-per-year cap is a hard business rule — it must never be bypassed.
- This job processes all pending referrals across all users — it must NOT leak data between users in its execution or logging.
- Log processing results at info level without including user IDs or financial amounts (use redemption counts only).
