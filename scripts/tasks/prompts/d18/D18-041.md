# Task D18-041: Security: scoping — physician can only see own referral code and credits

## What to Build

Create `apps/api/test/security/platform/referral.scoping.security.ts`.

Test that physicians can only access their own referral data. No physician should ever see another physician's referral code, credits, or redemption history.

### Cross-User Isolation Tests

**Setup:** Create two physicians (Physician A and Physician B), each with their own referral code and referral credits.

**Referral code isolation:**
- Physician A calls `GET /api/v1/referrals/my-code` → receives only Physician A's code
- Physician B calls `GET /api/v1/referrals/my-code` → receives only Physician B's code
- The response must NOT contain any reference to the other physician's code

**Referral credits isolation:**
- Physician A has 2 referral credits (1 QUALIFIED, 1 CREDITED)
- Physician B has 1 referral credit (PENDING)
- Physician A calls `GET /api/v1/referrals/my-credits` → sees only their 2 credits
- Physician B calls `GET /api/v1/referrals/my-credits` → sees only their 1 credit
- Neither response contains the other physician's credit data

**Credit application isolation:**
- Physician A has a QUALIFIED redemption
- Physician B attempts `POST /api/v1/referrals/{physicianA_redemptionId}/apply` → must return 403
- Physician A attempts the same → must succeed

**No enumeration via public endpoint:**
- `POST /api/v1/signup/referral-code` with a valid code returns `{ valid: true }` but does NOT reveal the code owner's identity, user ID, name, or any other information
- The response for a valid code is identical in structure to an invalid code response (only the `valid` field differs)

### Data Leakage Prevention

**Referral responses must NOT contain:**
- Other physicians' user IDs
- Other physicians' referral codes
- Other physicians' credit amounts or statuses
- Any patient data or claim data
- Any practice membership details (beyond what's needed for credit application)
- Stripe customer IDs or subscription IDs

**Summary counts must be scoped:**
- `GET /api/v1/referrals/my-credits` summary (total, pending, qualified, credited, expired) must only count the authenticated physician's credits

## Tests to Write

```typescript
describe('Referral Scoping Security', () => {
  describe('Referral code isolation', () => {
    it('physician A sees only their own referral code');
    it('physician B sees only their own referral code');
    it('my-code response does not contain other user data');
  });

  describe('Referral credits isolation', () => {
    it('physician A sees only their own credits');
    it('physician B sees only their own credits');
    it('credit count in summary is scoped to authenticated user');
    it('no cross-user credit data in response');
  });

  describe('Credit application isolation', () => {
    it('physician cannot apply another physician credit (403)');
    it('physician can apply their own credit (200)');
  });

  describe('Public endpoint does not leak identity', () => {
    it('valid code response does not contain owner user ID');
    it('valid code response does not contain owner name or email');
    it('valid and invalid code responses have identical structure');
  });

  describe('No sensitive data in referral responses', () => {
    it('my-credits response contains no Stripe IDs');
    it('my-credits response contains no patient or claim data');
    it('my-code response contains no subscription details');
  });
});
```

## Critical Security Rules

- Data scoping must be enforced at the **repository layer** (WHERE clause on `referrer_user_id`), not just the handler layer. Even if the handler passes the wrong user ID, the repository query must scope correctly.
- The public validation endpoint must return minimal data — only `{ valid: boolean }`. No additional fields.
- Error messages for cross-user access attempts must not reveal whether the resource exists. Use a generic 403 message like "Forbidden" rather than "Redemption belongs to another user."
