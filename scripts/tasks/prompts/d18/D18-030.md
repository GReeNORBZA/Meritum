# Task D18-030: Routes: referral endpoints + integration tests

## What to Build

Add referral routes to the platform domain router per Pricing Gap Closure Spec B3-5. Follow the existing routing pattern in `apps/api/src/domains/platform/` (likely Fastify route definitions with schema validation).

**Routes to implement:**

### GET /api/v1/referrals/my-code
- **Auth:** PHYSICIAN role required (authenticated session)
- **Handler:**
  1. Get `userId` from authenticated session
  2. Call `generateReferralCode(userId)` — this returns existing code or creates new one
  3. Return `{ referralCode: { referralCodeId, code, isActive, createdAt } }`
- **Validation:** `getMyReferralCodeSchema` from D18-004
- **Status:** 200

### GET /api/v1/referrals/my-credits
- **Auth:** PHYSICIAN role required
- **Handler:**
  1. Get `userId` from authenticated session
  2. Parse optional query params: `status`, `anniversaryYear`
  3. Call `findRedemptionsByReferrer(userId, { status, anniversaryYear })`
  4. Return `{ credits: [...], summary: { total, pending, qualified, credited, expired } }`
- **Validation:** `getMyReferralCreditsSchema` from D18-004
- **Status:** 200

### POST /api/v1/referrals/:redemptionId/apply
- **Auth:** PHYSICIAN role required
- **Handler:**
  1. Get `userId` from authenticated session
  2. Get `redemptionId` from params, `target` from body
  3. Call `applyReferralCredit(redemptionId, userId, target)`
  4. Return `{ redemption: { redemptionId, status, creditAppliedTo, creditAppliedAt } }`
- **Validation:** `applyReferralCreditSchema` from D18-004
- **Status:** 200

### POST /api/v1/signup/referral-code
- **Auth:** PUBLIC — no authentication required
- **Handler:**
  1. Get `code` from request body
  2. Call `findReferralCodeByCode(code)`
  3. If found and active: return `{ valid: true }` (200)
  4. If not found or inactive: return `{ valid: false }` (200) — do NOT return 404 to avoid code enumeration
- **Validation:** `validateReferralCodeSchema` from D18-004
- **Status:** 200 (always, even for invalid codes)

**Route registration:** Register these routes in the platform domain's route configuration. Follow the existing pattern — look at how other platform routes are registered (e.g., subscription routes, status page routes).

**Response format:** Follow the existing response envelope pattern in the codebase. Typically `{ data: {...} }` or direct object — match what other endpoints return.

## Critical Security Rules

- **Public endpoint security:** `POST /signup/referral-code` is public but must:
  - Rate limit aggressively (prevent code enumeration/brute force)
  - Always return 200 with `valid: true/false` — never 404 (prevents discovering valid codes)
  - Validate input strictly via Zod schema (8 chars, alphanumeric only)
- **Auth enforcement:** All other referral routes require authenticated PHYSICIAN session.
- **Scoping:** `my-code` and `my-credits` only return data for the authenticated user.
- **No PHI in responses:** Referral responses contain codes, statuses, and amounts — no patient data, no claim data.
