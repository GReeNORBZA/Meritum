# Task D18-021: Service: redeemReferralCode

## What to Build

Add the `redeemReferralCode` function to the referral service (in the same file as D18-020).

**Function signature:**
```typescript
redeemReferralCode(code: string, referredUserId: string): Promise<SelectReferralRedemption>
```

**Logic (Pricing Gap Closure Spec B3-2):**

1. **Validate code is active:**
   - Call `findReferralCodeByCode(code)`
   - If not found or `is_active === false`: throw `BusinessRuleError('Invalid or inactive referral code', { code: 'INVALID_REFERRAL_CODE' })`

2. **Validate referred user has never had a Meritum subscription:**
   - Query the `subscriptions` table for any record with `provider_id = referredUserId` (any status, including CANCELLED)
   - If any subscription exists: throw `BusinessRuleError('Referral codes are for new subscribers only', { code: 'REFERRED_NOT_NEW' })`
   - This ensures referrals are for genuinely new signups, not returning users

3. **Validate referrer and referred are NOT on the same practice:**
   - Query `practice_memberships` table (from Batch 1) for both `referrerUserId` (from the referral code record) and `referredUserId` where `is_active = true`
   - If both are active members of the same `practice_id`: throw `BusinessRuleError('Cannot refer a physician on the same practice', { code: 'SAME_PRACTICE_REFERRAL' })`
   - **Note:** This depends on Batch 1's `practice_memberships` table. If it doesn't exist yet, add a conditional check with a comment: `// Requires Batch 1 practice_memberships table`

4. **Validate referred user doesn't already have a pending redemption:**
   - Call `findPendingByReferredUser(referredUserId)`
   - If exists: throw `BusinessRuleError('Referral already pending for this user', { code: 'REFERRAL_ALREADY_PENDING' })`

5. **Calculate anniversary year:**
   - Look up the referrer's subscription `created_at` date
   - Calculate which anniversary year we're in: `Math.floor((now - subscriptionCreatedAt) / 365 days) + 1`
   - This is used for the 3-per-year cap (checked during qualification, not here)

6. **Create PENDING redemption:**
   - Call `createRedemption({ referralCodeId, referrerUserId, referredUserId, anniversaryYear })`
   - Return the created record

**Error handling:** All validation failures should use `BusinessRuleError` (or the project's equivalent error class). Check the existing codebase for the error class pattern used in `platform.service.ts`.

## Tests to Write

```typescript
describe('redeemReferralCode', () => {
  it('creates a PENDING redemption for a valid code and new user');
  it('rejects an invalid referral code');
  it('rejects an inactive (deactivated) referral code');
  it('rejects if referred user has an active subscription');
  it('rejects if referred user has a cancelled subscription (not new)');
  it('rejects if referrer and referred are on the same practice');
  it('allows referral if referrer and referred are on different practices');
  it('allows referral if neither is on a practice');
  it('rejects if referred user already has a pending redemption');
  it('correctly calculates the anniversary year for the referrer');
  it('returns the created redemption record with PENDING status');
});
```

## Critical Security Rules

- **Same-practice blocking is mandatory** per Spec B3-2: "Validates referrer and referred are NOT on the same practice." Practice seat additions are NOT referrals.
- The `referredUserId` must come from the authenticated session of the new user during signup, not from the referrer.
- The `code` input must be sanitized (Zod schema from D18-004 enforces `^[A-Z0-9]{8}$`).
- Never expose whether a code exists vs is inactive — use a generic error message for both cases.
