# Task D18-040: Security: authn + authz — referral routes authentication and authorization

## What to Build

Create `apps/api/test/security/platform/referral.authn.security.ts`.

Test that every authenticated referral route returns 401 without a valid session, and that only the correct users can access referral operations.

### Authentication Tests

**Protected routes — must return 401 without session:**

For each of these routes, send requests with: (a) no cookie, (b) expired cookie, (c) tampered cookie. All must return 401.

- `GET /api/v1/referrals/my-code`
- `GET /api/v1/referrals/my-credits`
- `POST /api/v1/referrals/:redemptionId/apply`

**Public route — must NOT require auth:**

- `POST /api/v1/signup/referral-code` → must return 200 (not 401) when called without a session cookie

### Authorization Tests

**Credit application ownership:**
- Physician A creates a referral, the referral qualifies → redemption belongs to Physician A
- Physician B (different user, valid session) calls `POST /api/v1/referrals/:redemptionId/apply` with Physician A's redemption ID → must return 403
- Physician A calls the same endpoint → must return 200

**Role enforcement:**
- A user without PHYSICIAN role (e.g., a PRACTICE_ADMIN-only user, or a DELEGATE) should get 403 on all referral routes
- Only PHYSICIAN role grants access to referral endpoints

### 401 Response Body Verification

- 401 responses must contain only an error object (e.g., `{ error: 'Unauthorized' }`)
- 401 responses must NOT contain any referral data, user IDs, or financial information
- 401 responses must NOT contain stack traces

## Tests to Write

```typescript
describe('Referral Authentication Security', () => {
  describe('GET /api/v1/referrals/my-code', () => {
    it('returns 401 with no session cookie');
    it('returns 401 with expired session cookie');
    it('returns 401 with tampered session cookie');
    it('returns 200 with valid PHYSICIAN session');
  });

  describe('GET /api/v1/referrals/my-credits', () => {
    it('returns 401 with no session cookie');
    it('returns 401 with expired session cookie');
    it('returns 401 with tampered session cookie');
    it('returns 200 with valid PHYSICIAN session');
  });

  describe('POST /api/v1/referrals/:redemptionId/apply', () => {
    it('returns 401 with no session cookie');
    it('returns 401 with expired session cookie');
    it('returns 401 with tampered session cookie');
    it('returns 403 when different physician tries to apply another user credit');
    it('returns 200 when credit owner applies their own credit');
  });

  describe('POST /api/v1/signup/referral-code (public)', () => {
    it('returns 200 without any session cookie');
    it('does not return 401 for anonymous requests');
  });

  describe('Role enforcement', () => {
    it('returns 403 for DELEGATE role on GET /referrals/my-code');
    it('returns 403 for PRACTICE_ADMIN-only role on GET /referrals/my-credits');
  });

  describe('401 response body', () => {
    it('does not leak referral data in 401 response');
    it('does not contain stack traces in 401 response');
  });
});
```

## Critical Security Rules

- Authentication must be enforced by middleware, not by individual route handlers.
- The public validation endpoint (`POST /signup/referral-code`) must genuinely be accessible without auth.
- 401 responses must be identical regardless of whether the resource exists — no information leakage.
- The authorization check on credit application (referrer_user_id matches session user) must be at the service layer, not just the route layer.
