# Task D16-021: Service: invitePhysician — validate PRACTICE_ADMIN, check not on practice/another practice, create invitation, emit notification

## What to Build

Add the `invitePhysician()` function to `apps/api/src/domains/platform/practice.service.ts`. Per Pricing Gap Closure Spec B1-7.

**Function signature:**
```typescript
async function invitePhysician(
  practiceId: string,
  email: string,
  invitedByUserId: string,
): Promise<SelectPracticeInvitation>
```

**Business logic:**

1. **Validate the inviter is PRACTICE_ADMIN for this practice:**
   - Find the practice by `practiceId`
   - Verify `practice.admin_user_id === invitedByUserId`
   - If not, throw `ForbiddenError` with code `NOT_PRACTICE_ADMIN`

2. **Validate the practice is ACTIVE:**
   - If `practice.status !== 'ACTIVE'`, throw `BusinessRuleError` with code `PRACTICE_NOT_ACTIVE`

3. **Normalize the email:**
   - `email.toLowerCase().trim()`

4. **Validate the email is not already on THIS practice:**
   - Look up the user by email
   - If user exists, call `membershipRepo.findMembershipByPracticeAndPhysician(practiceId, userId)`
   - If active membership found, throw `BusinessRuleError` with code `ALREADY_ON_PRACTICE`

5. **Validate the email is not already on ANOTHER active practice:**
   - If user exists, call `membershipRepo.findActiveMembershipByPhysicianId(userId)`
   - If found on a different practice, throw `BusinessRuleError` with code `ON_ANOTHER_PRACTICE`

6. **Check for existing pending invitation:**
   - Call `invitationRepo.findPendingInvitationByEmail(email, practiceId)`
   - If found, throw `BusinessRuleError` with code `INVITATION_ALREADY_PENDING`

7. **Generate invitation token:**
   - Generate a cryptographically random token (e.g., 32 bytes, hex-encoded = 64 chars)
   - Use `crypto.randomBytes(32).toString('hex')`
   - Hash the token: `crypto.createHash('sha256').update(token).digest('hex')`

8. **Create the invitation record:**
   - `practice_id`: practiceId
   - `invited_email`: normalized email
   - `invited_by_user_id`: invitedByUserId
   - `status`: `'PENDING'`
   - `token_hash`: the SHA-256 hash of the token
   - `expires_at`: `now + 7 days` (use `PRACTICE_INVITATION_EXPIRY_DAYS` constant)

9. **Emit notification:**
   - Send an invitation email to the invited physician
   - Email must include: practice name, inviter name, and the acceptance link containing the RAW token (not the hash)
   - Follow the existing notification pattern in the codebase

10. **Audit log:**
    - Log action: `'practice.invitation_sent'`
    - Include: practice ID, invited email, invited by user ID

11. **Return the invitation record** (without the raw token — the raw token is only sent via email).

## Critical Security Rules

- The raw invitation token is NEVER stored in the database. Only the SHA-256 hash is stored.
- The raw token is sent to the invited physician via email only.
- The invitation email must NOT contain any PHI about practice members or billing details.
- Email normalization (lowercase, trim) prevents duplicate invitations via case differences.
