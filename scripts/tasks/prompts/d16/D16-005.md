# Task D16-005: Add practice_id FK to subscriptions table, add PRACTICE_ADMIN role and permissions to IAM constants

## What to Build

This task modifies two files. Per Pricing Gap Closure Spec B1-4 and B1-6.

### Part 1: Add `practice_id` FK to subscriptions table

**File:** `packages/shared/src/schemas/db/platform.schema.ts`

Add a new column to the existing `subscriptions` table definition:

```typescript
practiceId: uuid('practice_id').references(() => practices.practiceId),
```

This column is NULLABLE. When set, this subscription is tied to a practice (clinic-tier). When null, it is an individual subscription.

Add an index for the new column in the table's index array:
```typescript
index('subscriptions_practice_id_idx').on(table.practiceId),
```

### Part 2: Add PRACTICE_ADMIN role and practice permissions to IAM constants

**File:** `packages/shared/src/constants/iam.constants.ts`

**Add to `Role` enum:**
```typescript
PRACTICE_ADMIN: 'PRACTICE_ADMIN',
```

**Add new permission keys to `Permission` enum:**
```typescript
// Practice management
PRACTICE_SEAT_VIEW: 'PRACTICE_SEAT_VIEW',
PRACTICE_SEAT_MANAGE: 'PRACTICE_SEAT_MANAGE',
PRACTICE_INVOICE_VIEW: 'PRACTICE_INVOICE_VIEW',
PRACTICE_SETTINGS_EDIT: 'PRACTICE_SETTINGS_EDIT',
```

**Add PRACTICE_ADMIN default permissions to `DefaultPermissions`:**
```typescript
[Role.PRACTICE_ADMIN]: [
  Permission.PRACTICE_SEAT_VIEW,
  Permission.PRACTICE_SEAT_MANAGE,
  Permission.PRACTICE_INVOICE_VIEW,
  Permission.PRACTICE_SETTINGS_EDIT,
] as readonly Permission[],
```

**PRACTICE_ADMIN role does NOT get any of these permissions:**
- Any `CLAIM_*` permissions
- Any `PATIENT_*` permissions
- Any `ANALYTICS_*` permissions
- `ADMIN_PHI_ACCESS`
- `REPORT_VIEW`, `REPORT_EXPORT`
- `AI_COACH_VIEW`, `AI_COACH_MANAGE`
- `SUBSCRIPTION_MANAGE` (individual subscription — practice admin manages practice billing, not individual)

This is non-negotiable per the pricing strategy document Section 2.4: "Individual physician billing data is personal professional information. Exposing it to a practice administrator would create a surveillance dynamic."

The practice admin CAN also be a physician with their own PHYSICIAN role. But the PRACTICE_ADMIN role itself is scoped strictly to seat list + invoice. Role stacking is handled by the IAM system — a user can have both PHYSICIAN and PRACTICE_ADMIN roles simultaneously.

**Update the `DefaultPermissions` type constraint.** The `satisfies Record<Role, readonly Permission[]>` constraint must still pass. Since `Role` type now includes `PRACTICE_ADMIN`, the `DefaultPermissions` object MUST include a `PRACTICE_ADMIN` entry.

## Critical Security Rules

- **DATA ISOLATION IS NON-NEGOTIABLE.** The PRACTICE_ADMIN role has ZERO access to claim data, patient data, analytics, AI coach, or any PHI. The only permissions are the 4 practice-specific ones: PRACTICE_SEAT_VIEW, PRACTICE_SEAT_MANAGE, PRACTICE_INVOICE_VIEW, PRACTICE_SETTINGS_EDIT.
- The practice admin sees the practice's consolidated invoice, NOT individual physician payment history.
- A user can hold both PHYSICIAN and PRACTICE_ADMIN roles. The PHYSICIAN role grants clinical access. The PRACTICE_ADMIN role grants practice management access. They are independent.
