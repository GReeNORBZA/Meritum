# Task D16-004: Drizzle schema for practice_invitations table

## What to Build

Add the `practiceInvitations` table to `packages/shared/src/schemas/db/platform.schema.ts`. Per Pricing Gap Closure Spec B1-3.

**Table definition:**

```typescript
export const practiceInvitations = pgTable(
  'practice_invitations',
  {
    invitationId: uuid('invitation_id').primaryKey().defaultRandom(),
    practiceId: uuid('practice_id')
      .notNull()
      .references(() => practices.practiceId),
    invitedEmail: varchar('invited_email', { length: 255 }).notNull(),
    invitedByUserId: uuid('invited_by_user_id')
      .notNull()
      .references(() => users.userId),
    status: varchar('status', { length: 20 }).notNull().default('PENDING'),
    tokenHash: varchar('token_hash', { length: 128 }).notNull(),
    expiresAt: timestamp('expires_at', { withTimezone: true }).notNull(),
    createdAt: timestamp('created_at', { withTimezone: true })
      .notNull()
      .defaultNow(),
  },
  (table) => [
    index('practice_invitations_practice_id_idx').on(table.practiceId),
    index('practice_invitations_token_hash_idx').on(table.tokenHash),
    index('practice_invitations_invited_email_idx').on(table.invitedEmail),
    index('practice_invitations_status_idx').on(table.status),
  ],
);
```

**Column details:**
| Column | Type | Nullable | Description |
|--------|------|----------|-------------|
| invitation_id | UUID PK | No | Primary key, auto-generated |
| practice_id | UUID FK -> practices | No | Inviting practice |
| invited_email | VARCHAR(255) | No | Email of physician being invited |
| invited_by_user_id | UUID FK -> users | No | Who sent the invitation |
| status | VARCHAR(20) | No | `'PENDING'`, `'ACCEPTED'`, `'DECLINED'`, `'EXPIRED'` — default `'PENDING'` |
| token_hash | VARCHAR(128) | No | Hashed invitation token (SHA-256 hex) |
| expires_at | TIMESTAMPTZ | No | Default: 7 days from creation (set by service layer) |
| created_at | TIMESTAMPTZ | No | Auto-set to now |

**Indexes:**
1. `practice_invitations_practice_id_idx` — for listing invitations per practice
2. `practice_invitations_token_hash_idx` — for fast token lookup on accept/decline
3. `practice_invitations_invited_email_idx` — for checking if email already invited
4. `practice_invitations_status_idx` — for filtering by status

## Critical Security Rules

- The `token_hash` column stores a SHA-256 hash of the invitation token, NEVER the raw token. The raw token is sent to the invited physician via email and never stored.
- The `invited_email` is the only physician-identifying data in this table. It is needed for the invitation workflow.
