# Task D16-030: Practice routes — all 8 endpoints for practice management

## What to Build

Create `apps/api/src/domains/platform/practice.routes.ts` with all practice management routes. Per Pricing Gap Closure Spec B1-8.

**Route definitions:**

| Method | Path | Auth | Zod Schema | Handler |
|--------|------|------|------------|---------|
| POST | /api/v1/practices | PHYSICIAN | createPracticeSchema | createPracticeHandler |
| GET | /api/v1/practices/:id | PRACTICE_ADMIN | practiceIdParamsSchema | getPracticeHandler |
| PATCH | /api/v1/practices/:id | PRACTICE_ADMIN | practiceIdParamsSchema + updatePracticeSchema | updatePracticeHandler |
| GET | /api/v1/practices/:id/seats | PRACTICE_ADMIN | practiceIdParamsSchema + practiceSeatsQuerySchema | getPracticeSeatsHandler |
| POST | /api/v1/practices/:id/invitations | PRACTICE_ADMIN | practiceIdParamsSchema + invitePhysicianSchema | invitePhysicianHandler |
| POST | /api/v1/practice-invitations/:token/accept | PHYSICIAN | acceptInvitationParamsSchema | acceptInvitationHandler |
| DELETE | /api/v1/practices/:id/seats/:userId | PRACTICE_ADMIN | removeSeatParamsSchema | removePhysicianHandler |
| GET | /api/v1/practices/:id/invoices | PRACTICE_ADMIN | practiceIdParamsSchema + practiceInvoicesQuerySchema | getPracticeInvoicesHandler |

**Route registration:**

Follow the existing route registration pattern in the codebase (likely Fastify or Express). Register all routes under a practice plugin/router.

**Authentication middleware:**
- All routes require authentication (session cookie)
- POST `/practices` requires PHYSICIAN role (any physician can create a practice)
- POST `/practice-invitations/:token/accept` requires PHYSICIAN role (accepting physician)
- All other routes require PRACTICE_ADMIN role for the specific practice

**Authorization middleware:**
- For routes with `:id` param (practice ID), verify the authenticated user is the `admin_user_id` of that practice
- Use the existing authorization middleware pattern but add practice-scoped checks

**Request validation:**
- Use the Zod schemas from `packages/shared/src/schemas/platform.schema.ts`
- Validate params, body, and query as appropriate
- Return 400 with validation errors for invalid input

**Response format:**
- Follow the existing API response format in the codebase
- Successful responses return the appropriate data
- Error responses follow the existing error format

## Critical Security Rules

- Every route MUST require authentication. No practice route is public.
- PRACTICE_ADMIN authorization must be per-practice — admin of practice A cannot access practice B.
- The accept invitation route uses PHYSICIAN auth, not PRACTICE_ADMIN (the accepting physician is not yet an admin).
- Request body validation via Zod prevents malformed input from reaching the service layer.
- **DATA ISOLATION:** The seats endpoint handler must NOT add any extra fields beyond what `getPracticeSeats()` returns. The response is: `physicianName`, `email`, `joinedAt`, `billingMode`. Nothing else.
