# Task D16-042: Security: scoping — MOST CRITICAL — practice admin NEVER sees PHI, claim data, billing volumes, rejection rates

## What to Build

Create `apps/api/test/security/platform/practice.scoping.security.ts`. Per Pricing Gap Closure Spec B1-10 and B1-11.

**THIS IS THE MOST CRITICAL SECURITY TEST IN THE ENTIRE CLINIC TIER.** It verifies the data isolation requirement that is non-negotiable per the pricing strategy document Section 2.4.

**Test setup:**
1. Create a practice with 5+ physicians
2. Some physicians should have claims, patients, analytics, AI suggestions, and payment history in the database
3. The practice admin queries the seats and invoice endpoints
4. Verify the responses contain ABSOLUTELY NOTHING from clinical/PHI tables

## Critical Security Rules

**THIS IS NON-NEGOTIABLE.**

Per Pricing Gap Closure Spec B1-11:

1. `GET /api/v1/practices/:id/seats` returns ONLY: `{ physicianName, email, joinedAt, billingMode }` per seat. **EXACTLY 4 fields per seat object. No more.**
2. No practice admin endpoint returns data from: `claims`, `ahcip_claim_details`, `wcb_claim_details`, `patients`, `analytics_cache`, `generated_reports`, `ai_suggestion_events`, `payment_history` (individual physician payments).
3. The practice admin sees the PRACTICE invoice (consolidated), not individual physician payment records.
4. **This is enforced at the repository layer, not the handler layer.** The SQL query for seats joins ONLY `practice_memberships` with `users`. No other tables.
5. Per the pricing strategy document Section 2.4: "Individual physician billing data is personal professional information. Exposing it to a practice administrator would create a surveillance dynamic."

The `billingMode` field IS allowed because it is practice account management data (which billing method is used for each seat), not individual financial data. It does not reveal the actual dollar amount.
