# Task D16-044: Security: leakage — error responses don't leak practice/member details to non-admins

## What to Build

Create `apps/api/test/security/platform/practice.leakage.security.ts` with data leakage prevention tests. Per Pricing Gap Closure Spec B1-10.

**Test setup:**
- Create two practices (A and B) with members
- Create authenticated sessions for: admin A, admin B, physician on A (non-admin), physician not on any practice

### Error Response Leakage Tests

```typescript
describe('Error Response Data Leakage Prevention', () => {
  describe('404 responses', () => {
    it('GET /practices/:nonExistentId returns generic 404 without revealing practice count');
    it('GET /practices/:id/seats for non-existent practice returns same error as forbidden');
    it('DELETE /practices/:id/seats/:nonExistentUserId returns generic error without revealing member list');
  });

  describe('403 responses', () => {
    it('GET /practices/:practiceB (as adminA) returns 403 without practice B name');
    it('GET /practices/:practiceB/seats (as adminA) returns 403 without member names');
    it('GET /practices/:practiceB/invoices (as adminA) returns 403 without billing details');
    it('POST /practices/:practiceB/invitations (as adminA) returns 403 without practice details');
    it('403 response body is identical regardless of which practice is accessed');
  });

  describe('422 responses', () => {
    it('invitation to already-on-practice email returns error without revealing which practice');
    it('invitation to email on another practice returns error without revealing other practice name');
    it('invitation to email on another practice returns error without revealing other practice admin');
    it('create practice when already on one returns error without revealing current practice details');
  });

  describe('Stripe ID leakage', () => {
    it('no error response contains stripe_customer_id');
    it('no error response contains stripe_subscription_id');
    it('no error response contains stripe_invoice_id');
    it('no success response for non-admin users contains Stripe IDs');
  });

  describe('Internal ID leakage', () => {
    it('no error response to non-admin contains practice_id of other practices');
    it('no error response to non-admin contains physician_user_id');
    it('no error response to non-admin contains membership_id');
    it('no error response to non-admin contains invitation_id of other practices');
  });
});
```

### Timing Attack Prevention

```typescript
describe('Timing Attack Prevention', () => {
  it('403 for non-existent practice responds in similar time as existing practice');
  it('404 response time is consistent regardless of resource existence');
});
```

### Stack Trace and Debug Info Leakage

```typescript

  it('no error response contains SQL query text');
  it('no error response contains database column names');
  it('no error response contains environment variable names');

```

## Critical Security Rules

- Error responses must be generic and consistent. Never vary the error message based on internal state.
- 403 and 404 responses must not differ in structure or content based on resource existence.
- Stripe IDs are internal billing references and must NEVER appear in client-facing responses (except to the practice admin for their own practice's management).
- Stack traces, file paths, SQL queries, and database column names must NEVER appear in any response.
- The error messages for "email already on this practice" vs "email on another practice" must not reveal the name or ID of the other practice.
