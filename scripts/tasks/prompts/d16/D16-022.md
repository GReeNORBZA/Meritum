# Task D16-022: Service: acceptInvitation — validate token, determine billing_mode, create membership, handle subscription transition

## What to Build

Add the `acceptInvitation()` function to `apps/api/src/domains/platform/practice.service.ts`. Per Pricing Gap Closure Spec B1-7.

**Function signature:**
```typescript
async function acceptInvitation(
  token: string,
  acceptingUserId: string,
): Promise<SelectPracticeMembership>
```

**Business logic:**

1. **Hash the incoming token:**
   - `const tokenHash = crypto.createHash('sha256').update(token).digest('hex')`

2. **Find the invitation by token hash:**
   - Call `invitationRepo.findInvitationByTokenHash(tokenHash)`
   - If not found, throw `NotFoundError` with code `INVITATION_NOT_FOUND`

3. **Validate the invitation is PENDING:**
   - If `invitation.status !== 'PENDING'`, throw `BusinessRuleError` with code `INVITATION_NOT_PENDING`

4. **Validate not expired:**
   - If `invitation.expires_at < now`, update status to EXPIRED and throw `BusinessRuleError` with code `INVITATION_EXPIRED`

5. **Validate the accepting user's email matches the invitation:**
   - Look up user by `acceptingUserId`
   - If `user.email.toLowerCase() !== invitation.invited_email.toLowerCase()`, throw `ForbiddenError` with code `EMAIL_MISMATCH`

6. **Validate the physician is not already on another practice:**
   - Call `membershipRepo.findActiveMembershipByPhysicianId(acceptingUserId)`
   - If found, throw `BusinessRuleError` with code `ALREADY_ON_PRACTICE`

7. **Determine billing_mode (CRITICAL LOGIC):**
   - Check if the physician has an active early bird subscription:
     - Query subscriptions where `provider_id = acceptingUserId` AND `plan IN ('EARLY_BIRD_MONTHLY', 'EARLY_BIRD_ANNUAL')` AND `status = 'ACTIVE'`
   - **If active early bird exists:** Set `billingMode = 'INDIVIDUAL_EARLY_BIRD'`
     - Physician keeps their individual early bird subscription
     - Practice Stripe subscription quantity is NOT incremented
   - **If NO active early bird:** Set `billingMode = 'PRACTICE_CONSOLIDATED'`
     - Cancel the physician's individual subscription (if any) via Stripe API
     - Increment the practice's Stripe subscription quantity by 1

8. **Create the membership (within a transaction):**
   - `practice_id`: invitation.practice_id
   - `physician_user_id`: acceptingUserId
   - `billing_mode`: determined in step 7
   - `joined_at`: now
   - `is_active`: true

9. **Update invitation status to ACCEPTED:**
   - Call `invitationRepo.updateInvitationStatus(invitation.invitation_id, 'ACCEPTED')`

10. **Handle Stripe subscription transition (within same transaction context):**
    - If `billingMode === 'PRACTICE_CONSOLIDATED'`:
      - If physician had an individual subscription: cancel it via `stripe.subscriptions.cancel()`
      - Update practice Stripe subscription: `stripe.subscriptions.update(practiceStripeSubId, { quantity: newCount, proration_behavior: 'create_prorations' })`
    - If `billingMode === 'INDIVIDUAL_EARLY_BIRD'`:
      - No Stripe changes needed — physician keeps their existing individual subscription

11. **Audit log:**
    - Log action: `'practice.invitation_accepted'`
    - Include: practice ID, physician user ID, billing mode, invitation ID

12. **Return the created membership record.**

## Critical Security Rules

- Token validation: hash the incoming token and compare against stored hash. NEVER compare raw tokens.
- Billing mode determination must happen at acceptance time, not invitation time. The physician's subscription status may change between invitation and acceptance.
- Stripe subscription quantity only counts `PRACTICE_CONSOLIDATED` members. Early bird members are excluded from the practice Stripe subscription.
- Transaction ensures atomicity — partial acceptance is not allowed. If Stripe call fails, roll back membership creation.
