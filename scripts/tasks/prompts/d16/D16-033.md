# Task D16-033: Practice Stripe integration — create subscription with quantity, update on member add/remove/transition, webhooks

## What to Build

Add practice-specific Stripe integration to `apps/api/src/domains/platform/practice.service.ts` (or create `apps/api/src/domains/platform/practice-stripe.service.ts` if separation is cleaner). Per Pricing Gap Closure Spec B1-9.

**New environment variables:**
- `STRIPE_PRICE_CLINIC_MONTHLY` — Stripe Price ID for clinic monthly plan
- `STRIPE_PRICE_CLINIC_ANNUAL` — Stripe Price ID for clinic annual plan

Add these to the env validation schema (follow existing pattern for `STRIPE_PRICE_STANDARD_MONTHLY` etc.).

### Functions to implement:

**createPracticeStripeSubscription(practice: SelectPractice, consolidatedSeatCount: number): Promise\<{ stripeCustomerId: string, stripeSubscriptionId: string }\>**

1. Create Stripe Customer (if not already created):
   - `stripe.customers.create({ name: practice.name, email: adminEmail, metadata: { practice_id: practice.practiceId } })`
   - NO PHI sent to Stripe

2. Determine price ID:
   - `practice.billingFrequency === 'MONTHLY'` ? `STRIPE_PRICE_CLINIC_MONTHLY` : `STRIPE_PRICE_CLINIC_ANNUAL`

3. Create Stripe Subscription:
   ```typescript
   stripe.subscriptions.create({
     customer: stripeCustomerId,
     items: [{ price: priceId, quantity: consolidatedSeatCount }],
     metadata: { practice_id: practice.practiceId },
   })
   ```
   - `quantity` = number of `PRACTICE_CONSOLIDATED` members only (NOT total headcount)
   - A practice could have quantity 0 if all members are on early bird

4. Update practice record with Stripe IDs

**updatePracticeStripeQuantity(practiceId: string, newQuantity: number): Promise\<void\>**

1. Find practice and get `stripe_subscription_id`
2. Update Stripe subscription:
   ```typescript
   stripe.subscriptions.update(stripeSubscriptionId, {
     items: [{ id: subscriptionItemId, quantity: newQuantity }],
     proration_behavior: 'create_prorations',
   })
   ```
3. Proration: Stripe automatically calculates prorated charges/credits

**cancelPracticeStripeSubscription(practiceId: string): Promise\<void\>**

1. Find practice and get `stripe_subscription_id`
2. Cancel: `stripe.subscriptions.cancel(stripeSubscriptionId)`
3. Update practice record: clear stripe_subscription_id, set status to CANCELLED

**handlePracticeStripeWebhook(event: Stripe.Event): Promise\<void\>**

Handle these events for practice subscriptions (identified by `metadata.practice_id`):

1. `invoice.created` — Add GST line item (5% of subtotal), same as individual subscriptions
2. `invoice.paid` — Update practice `current_period_start` and `current_period_end`
3. `invoice.payment_failed` — Follow existing dunning pattern but at practice level
4. `customer.subscription.updated` — Sync quantity and status changes
5. `customer.subscription.deleted` — Mark practice as CANCELLED

**Distinguish practice vs individual webhooks:**
- Check `event.data.object.metadata.practice_id`
- If present, route to practice webhook handler
- If absent, route to existing individual webhook handler

### Stripe quantity rules (critical):

- Stripe subscription quantity = count of `PRACTICE_CONSOLIDATED` members ONLY
- `INDIVIDUAL_EARLY_BIRD` members are NOT included in Stripe quantity
- When a member is added as `PRACTICE_CONSOLIDATED`: quantity++
- When a member is removed (was `PRACTICE_CONSOLIDATED`): quantity-- at end of month
- When an early bird transitions to `PRACTICE_CONSOLIDATED`: quantity++ (handled in Batch 2 but wire the hook now)
- A practice could have 7 total members but Stripe quantity of 4 (if 3 are early bird)

### Important: The 5-physician minimum for clinic tier eligibility counts ALL active members (both billing modes). But Stripe subscription quantity only counts `PRACTICE_CONSOLIDATED` members.

## Critical Security Rules

- **ZERO PHI sent to Stripe.** Only practice name and admin email in the Stripe customer. No patient names, claim data, or medical information.
- Stripe secret key from environment variable — never hardcoded or logged.
- Webhook signature verification must be applied to practice events just like individual events.
- Proration behavior ensures fair billing when members are added/removed mid-cycle.
- The distinction between total headcount (for minimum check) and Stripe quantity (for billing) is critical and must be tested.
