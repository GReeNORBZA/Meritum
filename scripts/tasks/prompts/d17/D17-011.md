# Task D17-011: Service — Early bird re-signup prevention

## What to Build

Modify `createCheckoutSession()` in `apps/api/src/domains/platform/platform.service.ts` to prevent users who have ever had an early bird subscription from signing up for early bird again — even after cancellation.

**Add to `createCheckoutSession()`, before the existing early bird cap check:**

```typescript

if (isEarlyBirdRate(plan)) {
  const hadEarlyBird = await repository.hasEverHadEarlyBird(userId);
  if (hadEarlyBird) {

      'Early bird rate does not survive cancellation',

```

**New repository method** in `apps/api/src/domains/platform/platform.repository.ts`:

```typescript
/**
 * Check if a user has EVER had an early bird subscription, including cancelled ones.
 * This prevents re-signup at early bird rates after cancellation.
 * Spec reference: B2-3 — "Early bird rate does not survive cancellation."
 */
async hasEverHadEarlyBird(userId: string): Promise<boolean> {
  // Query: SELECT EXISTS(
  //   SELECT 1 FROM subscriptions
  //   WHERE provider_id = userId
  //   AND plan IN ('EARLY_BIRD_MONTHLY', 'EARLY_BIRD_ANNUAL')
  // )
  // Note: no status filter — we check ALL subscriptions including CANCELLED
}
```

**Business rules (from spec B2-3):**
- When the plan is `EARLY_BIRD_MONTHLY` or `EARLY_BIRD_ANNUAL`, check if the user has ANY previous subscription with an early bird plan
- This includes subscriptions in ANY status: ACTIVE, CANCELLED, SUSPENDED, etc.
- If they have ever had early bird, reject with error code `EARLY_BIRD_INELIGIBLE`
- This check runs BEFORE the 100-spot cap check — a returning user should not consume a cap slot
- The error message should be clear: "Early bird rate does not survive cancellation"

**Order of checks in createCheckoutSession() for early bird plans:**
1. `hasEverHadEarlyBird(userId)` → reject if true (D17-011, this task)
2. `countEarlyBirdSubscriptions() < EARLY_BIRD_CAP` → reject if at cap (existing logic)
3. Proceed with Stripe checkout creation

## Critical Security Rules

- This is a revenue protection measure — circumventing it would allow users to repeatedly get the $199 rate.
- The check must be server-side and cannot be bypassed by client manipulation.
- The query must check ALL subscription statuses, not just ACTIVE — a WHERE clause filtering only active subs would be a vulnerability.
- Race condition: if two concurrent requests try to sign up, the early bird cap check uses a transaction or SELECT FOR UPDATE. The re-signup check does not need this since it's a read-only existence check.
