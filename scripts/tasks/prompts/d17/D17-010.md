# Task D17-010: Service — Early bird rate lock on checkout completion

## What to Build

Modify the Stripe webhook handler in `apps/api/src/domains/platform/platform.service.ts` to set the `early_bird_locked_until` timestamp when an early bird checkout is completed.

**Locate the `checkout.session.completed` webhook handler** (or the `handleCheckoutSessionCompleted` function). When the completed checkout session is for an early bird plan (`EARLY_BIRD_MONTHLY` or `EARLY_BIRD_ANNUAL`), add the following logic:

```typescript
// After creating/updating the subscription record:
if (isEarlyBirdRate(plan)) {
  const lockedUntil = new Date(createdAt);
  lockedUntil.setMonth(lockedUntil.getMonth() + EARLY_BIRD_RATE_LOCK_MONTHS); // 12 months

  await repository.updateSubscription(subscriptionId, {
    earlyBirdLockedUntil: lockedUntil,
  });
}
```

**Import required constants and utils:**

```typescript
import { EARLY_BIRD_RATE_LOCK_MONTHS } from '@meritum/shared/constants/platform.constants.js';
import { isEarlyBirdRate } from '@meritum/shared/utils/pricing.utils.js';
```

**Repository method needed:** Ensure `updateSubscription()` in `apps/api/src/domains/platform/platform.repository.ts` can accept and persist the `earlyBirdLockedUntil` field. If the method already uses a generic update pattern (spread of columns), it should work automatically. If it uses an explicit column list, add `earlyBirdLockedUntil` to the allowed fields.

**Business rules (from spec B2-2):**
- `early_bird_locked_until = created_at + 12 months` — this is the hard date when the early bird rate expires
- Only set for plans that match `EARLY_BIRD_MONTHLY` or `EARLY_BIRD_ANNUAL`
- The lock is per-subscription, not per-user. If a user somehow had a second early bird subscription (which D17-011 prevents), each would have its own lock.
- The `earlyBirdExpiryNotified` flag remains `false` (default) — it is set later by the scheduled job in D17-012.

## Critical Security Rules

- The rate lock timestamp is immutable after creation — do not allow it to be extended or reset.
- Verify that the lock date calculation handles month-end edge cases correctly (e.g., Jan 31 + 12 months = Jan 31 next year, not Feb).
- Audit log the rate lock creation event.
