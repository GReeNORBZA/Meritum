# Task D17-040: Security — Early bird re-signup prevention after cancellation

## What to Build

Create `apps/api/test/security/platform/platform.earlybird-resignup.security.ts` to verify that the early bird re-signup prevention cannot be bypassed.

**Security test cases:**

```typescript
import { describe, it, expect, beforeAll, afterAll } from 'vitest';

describe('Security: Early Bird Re-Signup Prevention', () => {

  describe('Direct API bypass attempts', () => {
    it('POST /api/v1/subscriptions/checkout with EARLY_BIRD_MONTHLY plan returns 400 EARLY_BIRD_INELIGIBLE for user with cancelled early bird');
    it('POST /api/v1/subscriptions/checkout with EARLY_BIRD_ANNUAL plan returns 400 EARLY_BIRD_INELIGIBLE for user with cancelled early bird');
    it('request body manipulation: changing plan field after validation does not bypass check');
  });

  describe('Status manipulation attempts', () => {
    it('user with CANCELLED early bird subscription cannot re-sign up');
    it('user with SUSPENDED early bird subscription cannot sign up for new early bird');
    it('user whose early bird expired and transitioned to STANDARD cannot sign up for early bird again');
    it('user whose early bird expired in a practice cannot sign up for early bird individually');
  });

  describe('Cross-plan attempts', () => {
    it('user with cancelled EARLY_BIRD_MONTHLY cannot sign up for EARLY_BIRD_ANNUAL');
    it('user with cancelled EARLY_BIRD_ANNUAL cannot sign up for EARLY_BIRD_MONTHLY');
    it('user with expired+transitioned EARLY_BIRD_MONTHLY cannot sign up for EARLY_BIRD_ANNUAL');
  });

  describe('Timing attacks', () => {
    it('concurrent early bird signup requests from same user: only first succeeds');
    it('re-signup attempt during the cancellation grace period is still blocked');
  });

  describe('Account manipulation', () => {
    it('user cannot bypass check by creating a new subscription record directly (API does not expose direct subscription creation)');
    it('error response does not reveal whether user had early bird before (generic message)');
  });

  describe('Positive cases (should succeed)', () => {
    it('user with cancelled early bird CAN sign up for STANDARD_MONTHLY');
    it('user with cancelled early bird CAN sign up for STANDARD_ANNUAL');
    it('user with cancelled early bird CAN sign up for CLINIC_MONTHLY (via practice)');
    it('user with cancelled early bird CAN sign up for CLINIC_ANNUAL (via practice)');
    it('user who has NEVER had early bird CAN sign up for EARLY_BIRD_MONTHLY');
    it('user who has NEVER had early bird CAN sign up for EARLY_BIRD_ANNUAL');
  });
});
```

**Test setup:**
- Create test users with various early bird subscription histories (active, cancelled, expired, transitioned)
- Use authenticated API requests (session cookies) to attempt checkout
- Mock Stripe to capture checkout session creation attempts
- Verify that the Stripe checkout session is NEVER created for blocked attempts

**What to verify:**
1. The `hasEverHadEarlyBird()` check runs before the early bird cap check
2. The check queries ALL subscription statuses, not just ACTIVE
3. The error response uses a generic message (does not leak subscription history)
4. The early bird cap is NOT decremented for rejected re-signup attempts

## Critical Security Rules

- This is a revenue protection security test — early bird re-signup bypass would cause financial loss.
- The error response must not leak information about the user's subscription history to potential attackers.
- The 401/400 distinction matters: unauthenticated requests get 401, authenticated but ineligible requests get 400 with EARLY_BIRD_INELIGIBLE code.
