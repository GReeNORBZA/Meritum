# Task D17-012: Service — checkEarlyBirdExpiry scheduled job

## What to Build

Create a scheduled job function `checkEarlyBirdExpiry()` in `apps/api/src/domains/platform/platform.service.ts` that handles early bird subscription expiry — both the 30-day warning and the actual transition.

**Function signature:**

```typescript
/**
 * Scheduled job: check for early bird subscriptions approaching or past expiry.
 * Runs daily via cron.
 *
 * Two phases:
 * 1. Warning phase: early_bird_locked_until <= now() + 30 days AND early_bird_expiry_notified = false
 * 2. Transition phase: early_bird_locked_until <= now()
 *
 * Spec reference: B2-2 — Early Bird Rate Lock
 */
export async function checkEarlyBirdExpiry(): Promise<void>
```

**Phase 1 — 30-day warning notification:**

```typescript
// Find subscriptions where:
// - plan IN ('EARLY_BIRD_MONTHLY', 'EARLY_BIRD_ANNUAL')
// - status = 'ACTIVE'
// - early_bird_locked_until IS NOT NULL
// - early_bird_locked_until <= now() + 30 days
// - early_bird_expiry_notified = false

// For each:
// 1. Emit EARLY_BIRD_EXPIRING notification to the physician
// 2. Set early_bird_expiry_notified = true
```

**New repository method:**

```typescript

// Returns active early bird subs where locked_until <= now() + warningDays AND not yet notified
```

**Phase 2 — Expiry transition:**

```typescript
// Find subscriptions where:
// - plan IN ('EARLY_BIRD_MONTHLY', 'EARLY_BIRD_ANNUAL')
// - status = 'ACTIVE'
// - early_bird_locked_until IS NOT NULL
// - early_bird_locked_until <= now()

// For each, determine the transition path:

// PATH A: Physician is in a practice (has active practice_memberships row)
// 1. Cancel the individual early bird Stripe subscription
// 2. Update subscription status to CANCELLED
// 3. Transition practice membership billing_mode from 'INDIVIDUAL_EARLY_BIRD' to 'PRACTICE_CONSOLIDATED'
// 4. Increment the practice's Stripe subscription quantity by 1
// 5. Emit EARLY_BIRD_EXPIRED notification to physician
// 6. Emit PRACTICE_MEMBER_TRANSITIONED notification to practice admin
// 7. Audit log the transition

// PATH B: Physician is NOT in a practice
// 1. Determine new plan based on current billing frequency:
//    - EARLY_BIRD_MONTHLY → STANDARD_MONTHLY
//    - EARLY_BIRD_ANNUAL → STANDARD_ANNUAL
// 2. Update Stripe subscription to the new price ID (stripe.subscriptions.update with price change)
// 3. Update subscription record: plan = new plan, early_bird_locked_until = null
// 4. Emit EARLY_BIRD_EXPIRED notification to physician with new rate info
// 5. Audit log the transition
```

**New repository methods:**


**Stripe operations:**
- Cancel subscription: `stripe.subscriptions.cancel(stripeSubscriptionId)`
- Update price: `stripe.subscriptions.update(stripeSubscriptionId, { items: [{ id: itemId, price: newPriceId }] })`
- Increment quantity: `stripe.subscriptions.update(stripeSubscriptionId, { items: [{ id: itemId, quantity: currentQuantity + 1 }] })`

**Price ID mapping for transitions:**
- `EARLY_BIRD_MONTHLY` → `STRIPE_PRICE_STANDARD_MONTHLY` (env var)
- `EARLY_BIRD_ANNUAL` → `STRIPE_PRICE_STANDARD_ANNUAL` (env var)

## Critical Security Rules

- The scheduled job must be idempotent — running it twice should not duplicate notifications or Stripe operations.
- Use database transactions for the transition operations to ensure atomicity (especially Path A where multiple tables and Stripe are updated).
- The Stripe cancellation and price change must happen before the database update — if Stripe fails, do not update the local record.
- Never log Stripe subscription IDs in user-facing error messages.
- The practice admin notification (Path A) must NOT include the physician's billing amount — only that a member has transitioned to consolidated billing.
