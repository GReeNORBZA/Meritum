# Task D17-015: Integration tests — Full early bird lifecycle

## What to Build

Create `apps/api/test/integration/platform/early-bird-lifecycle.integration.test.ts` with end-to-end integration tests that exercise the complete early bird lifecycle — from signup through rate lock, expiry, re-signup prevention, and practice transition.

These tests verify the interaction between multiple service functions and repository methods working together across the full lifecycle.

**Test scenarios:**

```typescript
import { describe, it, expect, beforeAll, afterAll, beforeEach } from 'vitest';

describe('Early Bird Lifecycle Integration Tests', () => {

  describe('Scenario 1: Individual early bird → standard transition', () => {
    it('physician signs up for EARLY_BIRD_MONTHLY');
    it('subscription has early_bird_locked_until set to 12 months from creation');
    it('early_bird_expiry_notified is false');
    it('at 30 days before expiry: checkEarlyBirdExpiry emits EARLY_BIRD_EXPIRING');
    it('early_bird_expiry_notified is now true');
    it('running checkEarlyBirdExpiry again does NOT re-notify');
    it('at expiry: checkEarlyBirdExpiry transitions to STANDARD_MONTHLY');
    it('Stripe subscription is updated to standard price');
    it('subscription record reflects new plan');
    it('physician receives EARLY_BIRD_EXPIRED notification');
  });

  describe('Scenario 2: Early bird → cancellation → re-signup blocked', () => {
    it('physician signs up for EARLY_BIRD_MONTHLY');
    it('physician cancels subscription');
    it('physician attempts EARLY_BIRD_MONTHLY signup → rejected with EARLY_BIRD_INELIGIBLE');
    it('physician attempts EARLY_BIRD_ANNUAL signup → rejected with EARLY_BIRD_INELIGIBLE');
    it('physician can sign up for STANDARD_MONTHLY (allowed)');
    it('physician can sign up for STANDARD_ANNUAL (allowed)');
  });

  describe('Scenario 3: Early bird physician joins practice', () => {
    it('physician on active early bird accepts practice invitation');
    it('membership billing_mode is INDIVIDUAL_EARLY_BIRD');
    it('practice Stripe quantity is NOT incremented');
    it('physician counts toward practice headcount');
    it('practice remains eligible for clinic tier (>= 5 members)');
    it('physician still has their individual early bird subscription');
  });

  describe('Scenario 4: Early bird expires while in practice', () => {
    it('physician on INDIVIDUAL_EARLY_BIRD in a practice');
    it('early bird rate lock expires');
    it('checkEarlyBirdExpiry: cancels individual subscription');
    it('checkEarlyBirdExpiry: transitions billing_mode to PRACTICE_CONSOLIDATED');
    it('checkEarlyBirdExpiry: increments practice Stripe quantity');
    it('physician notified of transition');
    it('practice admin notified of member transition');
    it('practice headcount unchanged');
  });

  describe('Scenario 5: Practice with all early bird members', () => {
    it('create practice where all 5 founding members are on early bird');
    it('practice Stripe subscription quantity = 0');
    it('practice still qualifies for clinic tier (headcount = 5)');
    it('as each early bird expires, quantity increments: 0→1→2→3→4→5');
    it('after last expiry: practice admin receives PRACTICE_ALL_MEMBERS_POST_EARLY_BIRD');
  });

  describe('Scenario 6: Mixed practice — some early bird, some standard', () => {
    it('practice has 3 early bird and 4 standard members (7 total)');
    it('practice Stripe quantity = 4 (only consolidated members)');
    it('as early birds expire, quantity goes 4→5→6→7');
    it('all members counted for headcount throughout');
  });

  describe('Scenario 7: Early bird annual lifecycle', () => {
    it('physician signs up for EARLY_BIRD_ANNUAL');
    it('subscription has early_bird_locked_until set to 12 months');
    it('at expiry: transitions to STANDARD_ANNUAL');
    it('Stripe subscription updated to STANDARD_ANNUAL price');
    it('after cancellation: re-signup for EARLY_BIRD_ANNUAL blocked');
    it('after cancellation: re-signup for EARLY_BIRD_MONTHLY also blocked');
  });
});
```

**Test setup requirements:**
- Use test helpers to create mock Stripe responses (following existing test patterns in the codebase)
- Use test database helpers to seed subscriptions, practices, and memberships
- Mock the notification emitter to capture and assert on emitted notifications
- Use time manipulation (fake timers or date mocking) to simulate the 12-month expiry period

**Look at existing integration tests** in `apps/api/test/integration/platform/` for patterns on:
- Test database setup and teardown
- Stripe mock configuration
- Authentication helpers
- Notification assertion helpers

## Critical Security Rules

- Integration tests must verify that early bird re-signup prevention works end-to-end (not just at the repository level).
- Tests must verify that practice admin notifications do NOT contain physician billing details.
- Tests must verify that the Stripe quantity is correct at each stage of the lifecycle.
