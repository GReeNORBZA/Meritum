# IMA-022: IMA Amendment — Handlers, Routes, and Blocking Gate Middleware

## What to Build

### 1. Amendment Handlers

Extend `apps/api/src/domains/platform/platform.handlers.ts` (or create a separate
handlers file) with:

- **`createAmendment`** — Admin only. Parse `createAmendmentSchema`, call service.
  Return 201 with the amendment.
- **`listAmendments`** — Admin only. Return paginated list of amendments.
- **`getAmendment`** — Admin only. Return single amendment with response summary.
- **`acknowledgeAmendment`** — Physician. Respond to a non-material amendment.
- **`respondToAmendment`** — Physician. Respond to a material amendment with
  ACCEPTED or REJECTED.
- **`getMyPendingAmendments`** — Physician. Return their unresponded amendments.

### 2. Routes

Register in `platform.routes.ts`:

| Method | Path | Access |
|--------|------|--------|
| `POST` | `/api/v1/platform/amendments` | Admin only |
| `GET` | `/api/v1/platform/amendments` | Admin only |
| `GET` | `/api/v1/platform/amendments/:id` | Admin only |
| `POST` | `/api/v1/platform/amendments/:id/acknowledge` | Physician |
| `POST` | `/api/v1/platform/amendments/:id/respond` | Physician |
| `GET` | `/api/v1/account/pending-amendments` | Physician (IAM routes) |

### 3. Gate Middleware

Extend `auth.plugin.ts` or create a separate middleware file:

- **`checkAmendmentGate`** — Pre-handler that calls
  `getBlockingAmendments(providerId)`. If any blocking amendments exist, return **403**
  with error code `IMA_AMENDMENT_REQUIRED` and a list of amendment IDs requiring
  acknowledgement.

- Apply this middleware to **all PHI-accessing routes** EXCEPT:
  - The amendment acknowledge/respond endpoints (prevents circular dependency)
  - The `/api/v1/account/*` endpoints
  - The logout endpoint
  - Data export endpoints (always available per IMA)

## Critical Security Rules

- Admin-only endpoints must verify `role === ADMIN` before processing.
- The gate middleware must **never** block the amendment acknowledgement or respond
  endpoints — doing so would create a circular dependency where the physician cannot
  acknowledge because the gate blocks them.
- The gate middleware must **never** block data export routes (physician right to their
  data under IMA).
- The `ip_address` and `user_agent` stored with responses must be extracted from the
  HTTP request object, not from the request body.
- Amendment `:id` path parameter must be validated as a UUID; reject with 400 otherwise.

## Tests to Write

```typescript
import { describe, it, expect } from 'vitest';

describe('Amendment Handlers and Gate Middleware', () => {
  it('POST /api/v1/platform/amendments creates amendment (admin only)', async () => {
    // Send POST as admin with valid amendment data
    // Expect 201 with amendment record
  });

  it('POST /api/v1/platform/amendments returns 403 for non-admin', async () => {
    // Send POST as physician
    // Expect 403
  });

  it('POST /api/v1/platform/amendments/:id/acknowledge records acknowledgement', async () => {
    // Create a NON_MATERIAL amendment as admin
    // Send POST as physician to acknowledge
    // Expect 200 with acknowledgement confirmation
  });

  it('gate middleware blocks PHI access when unacknowledged non-material amendment exists', async () => {
    // Create NON_MATERIAL amendment with effective_date in the past
    // Access a PHI endpoint as a physician who has not acknowledged
    // Expect 403 with error code IMA_AMENDMENT_REQUIRED
  });

  it('gate middleware does not block amendment acknowledgement endpoint', async () => {
    // With a blocking amendment active
    // POST to /amendments/:id/acknowledge
    // Expect 200, not 403
  });

  it('gate middleware does not block data export', async () => {
    // With a blocking amendment active
    // Access data export endpoint
    // Expect success, not 403
  });

  it('gate middleware does not block logout', async () => {
    // With a blocking amendment active
    // POST to logout endpoint
    // Expect success, not 403
  });
});
```
