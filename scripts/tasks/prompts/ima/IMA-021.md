# IMA-021: IMA Amendment — Service

## What to Build

Extend `apps/api/src/domains/platform/platform.service.ts` with amendment service
functions:

- **`createAmendment(adminCtx, data)`** — Admin-only. Create amendment record via
  repository. Emit `IMA_AMENDMENT_NOTICE` notification to all active physicians. Emit
  audit event `amendment.created`. Return the amendment.

- **`acknowledgeAmendment(ctx, amendmentId)`** — For NON_MATERIAL amendments. Create
  response with type `ACKNOWLEDGED`. Emit audit event `amendment.acknowledged`.

- **`respondToAmendment(ctx, amendmentId, responseType)`** — For MATERIAL amendments.
  Create response with type `ACCEPTED` or `REJECTED`. Emit audit event
  `amendment.accepted` or `amendment.rejected`.

- **`getBlockingAmendments(providerId)`** — Return list of NON_MATERIAL amendments past
  their `effective_date` that the provider hasn't acknowledged. Used by gate middleware.

- **`runAmendmentReminders()`** — Scheduled job: find MATERIAL amendments where the
  deadline is 30 days or 7 days away, emit `IMA_AMENDMENT_REMINDER` to providers who
  haven't responded yet.

### Gate Middleware Logic

- A **NON_MATERIAL** amendment blocks PHI access after its `effective_date` until
  acknowledged.
- A **MATERIAL** amendment does **NOT** block (silence = existing terms continue per
  IMA section 11.3).

## Critical Security Rules

- Only users with `ADMIN` role can call `createAmendment`. Enforce this check at the
  service layer as defense-in-depth (do not rely solely on route-level guards).
- Any authenticated physician can acknowledge or respond to amendments targeting them.
- Audit events must include the acting user ID, timestamp, and amendment ID.
- Notification delivery failures must not cause the create/acknowledge transaction to
  fail; use fire-and-forget or a queue.

## Prerequisites

- **IMA-020** — Amendment repository functions.
