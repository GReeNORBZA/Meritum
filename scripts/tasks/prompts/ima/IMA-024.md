# IMA-024: IMA Amendment — Security Tests

## What to Build

Create `apps/api/test/security/platform/platform.amendment.security.ts` with security
tests covering all five applicable categories.

### Category 1: Authentication (authn)

Verify that every amendment endpoint rejects unauthenticated requests.

- `POST /api/v1/platform/amendments` returns 401 without session
- `GET /api/v1/platform/amendments` returns 401 without session
- `GET /api/v1/platform/amendments/:id` returns 401 without session
- `POST /api/v1/platform/amendments/:id/acknowledge` returns 401 without session
- `POST /api/v1/platform/amendments/:id/respond` returns 401 without session
- `GET /api/v1/account/pending-amendments` returns 401 without session

### Category 2: Authorization (authz)

Verify role-based access is enforced correctly.

- `POST /api/v1/platform/amendments` returns 403 for physician role
- `POST /api/v1/platform/amendments` returns 403 for delegate role
- Acknowledge endpoint works for physician role
- Respond endpoint works for physician role
- Acknowledge endpoint returns 403 for delegate without physician context

### Category 3: Input Validation

Verify malicious or malformed input is rejected.

- SQL injection payloads in `title` and `description` are rejected
- XSS payloads in `title` and `description` are rejected or sanitized
- Non-UUID amendment ID in path returns 400
- Empty `document_text` is rejected with validation error
- Invalid `amendment_type` value is rejected with validation error

### Category 4: Leakage Prevention

Verify error responses do not expose sensitive internals.

- Error responses do not echo back submitted content
- Request for non-existent amendment returns 404 (not 500)
- Database errors are masked with a generic message

### Category 5: Audit Trail

Verify all state-changing operations produce audit records.

- Amendment creation produces an audit record with `amendment.created` event
- Acknowledgement produces an audit record with `amendment.acknowledged` event
- Accept response produces an audit record with `amendment.accepted` event
- Reject response produces an audit record with `amendment.rejected` event

## Critical Security Rules

- All tests must run against the actual HTTP layer (not bypassing middleware).
- SQL injection tests must use real payloads (e.g., `'; DROP TABLE--`, `1 OR 1=1`).
- XSS tests must use real payloads (e.g., `<script>alert(1)</script>`,
  `<img onerror=alert(1)>`).
- Audit trail tests must verify the audit record contains the correct actor, action,
  target, and timestamp.
- Never store test credentials in source code; use test fixtures or environment helpers.

## Prerequisites

- **IMA-023** — Integration tests are passing, confirming the full amendment lifecycle
  works correctly before layering security tests on top.
