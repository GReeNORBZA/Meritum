# IMA-023: IMA Amendment — Integration Tests

## What to Build

Create `apps/api/test/integration/platform/amendments.integration.ts` with full
lifecycle tests covering the complete amendment flow from creation through response.

### Test Structure

```typescript
describe('IMA Amendment Lifecycle', () => {
  // Setup: create admin user, create two physician users with active subscriptions

  describe('Admin creates non-material amendment', () => {
    it('creates amendment and returns 201');
    it('all active physicians receive IMA_AMENDMENT_NOTICE notification');
    it('amendment appears in admin list');
  });

  describe('Physician acknowledges non-material amendment', () => {
    it('before effective_date: PHI endpoints work normally');
    it('after effective_date: PHI endpoints return 403 IMA_AMENDMENT_REQUIRED');
    it('physician acknowledges: returns 200');
    it('after acknowledgement: PHI endpoints work again');
    it('double acknowledgement: returns 409');
  });

  describe('Admin creates material amendment', () => {
    it('creates amendment and returns 201');
    it('after effective_date: PHI endpoints still work (material does not block)');
    it('physician accepts: returns 200');
    it('physician rejects: returns 200');
    it('physician who did not respond: existing terms continue');
  });

  describe('Amendment reminders', () => {
    it('reminder sent at 30 days before deadline for material amendments');
    it('reminder sent at 7 days before deadline');
  });
});
```

### Key Scenarios to Cover

- **Non-material blocking**: Verify the gate middleware blocks PHI access after
  `effective_date` and unblocks after acknowledgement.
- **Material non-blocking**: Verify MATERIAL amendments never block PHI access
  regardless of whether the physician has responded.
- **Double response prevention**: Verify 409 Conflict when a physician tries to
  acknowledge or respond twice.
- **Notification delivery**: Verify `IMA_AMENDMENT_NOTICE` is emitted on creation
  and `IMA_AMENDMENT_REMINDER` is emitted by the scheduled job.
- **Multi-physician**: Use two physician users to confirm that one physician's
  acknowledgement does not affect the other's gate status.
- **Time-sensitive logic**: Use date mocking or test helpers to simulate
  `effective_date` transitions (before vs after).

## Critical Security Rules

- Tests must use proper authentication fixtures (admin session for admin endpoints,
  physician session for physician endpoints).
- Never hardcode real credentials or tokens in test files.
- Tests must clean up created records to avoid state leakage between test runs.
- Verify that error responses do not leak internal details (stack traces, SQL, etc.).

## Prerequisites

- **IMA-022** — Handlers, routes, and gate middleware are implemented and functional.
