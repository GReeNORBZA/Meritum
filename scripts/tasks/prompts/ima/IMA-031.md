# IMA-031: Breach notification — service

## What to Build

Extend platform service with breach functions:

- `createBreach(adminCtx, data)` — admin-only. Create breach record. Add all affected custodians from data.affected_provider_ids. Emit audit `breach.created`. Return breach.
- `sendBreachNotifications(adminCtx, breachId)` — find all unnotified custodians. For each: send BREACH_INITIAL_NOTIFICATION to their primary AND secondary email (if set). Mark as notified. Create a breach_update record of type INITIAL. Emit audit `breach.notification_sent`. Track the 72h deadline from awarenessDate.
- `addBreachUpdate(adminCtx, breachId, content)` — create SUPPLEMENTARY breach_update. Send BREACH_UPDATE notification to all affected custodians (both emails). Emit audit `breach.updated`.
- `resolveBreach(adminCtx, breachId)` — set status RESOLVED, set resolvedAt. Emit audit `breach.resolved`.
- `checkBreachDeadlines()` — scheduled job: find breaches where awarenessDate + 72h is approaching and custodians are still unnotified. Alert admin (internal notification or log warning).

Key: breach notifications MUST go to both primary and secondary email per IMA section 11.7. Use the notification service's dual-delivery mechanism added in IMA-016.

## Critical Security Rules

- Admin-only for all breach operations.
- No PHI in breach notifications sent to custodians — describe the breach in general terms.
- The affected_provider_ids list is never exposed to non-admin users.

## Prerequisites

- IMA-030 (repository)
- IMA-016 (dual-email delivery)
