# IMA-020: IMA Amendment — Repository

## What to Build

Extend `apps/api/src/domains/platform/platform.repository.ts` (or create a new file
alongside it) with amendment repository functions:

- **`createAmendment(data)`** — Insert into `ima_amendments`. Compute SHA-256 hash of
  `document_text` and store in `document_hash`. Set `notice_date` to `now()`. Return
  the amendment record.

- **`findAmendmentById(amendmentId)`** — Return amendment with response count summary.

- **`listAmendments(filters)`** — Paginated list with status derived from
  `effective_date` vs `now`.

- **`findPendingAmendmentsForProvider(providerId)`** — Return all amendments where
  `effective_date` has passed and this provider has no response in
  `ima_amendment_responses`. Used by the gate middleware.

- **`createAmendmentResponse(data)`** — Insert into `ima_amendment_responses` with
  `response_type`, `ip_address`, `user_agent`. Enforce
  `unique(amendment_id, provider_id)` — return conflict error if already responded.

- **`getAmendmentResponse(amendmentId, providerId)`** — Check if provider has responded.

- **`countUnrespondedAmendments(providerId)`** — Count amendments past `effective_date`
  with no response. Used for the gate check.

## Critical Security Rules

- The `ip_address` and `user_agent` for a response MUST come from the HTTP request
  context, never from the client request body.
- The `document_hash` MUST be computed server-side from `document_text` using SHA-256
  (import `createHash` from Node `crypto`). Never accept a client-provided hash.
- All IDs used in queries must be validated as UUIDs before reaching the database layer.

## Prerequisites

- **IMA-002** — Database schema definitions for `ima_amendments` and
  `ima_amendment_responses` tables.
- **IMA-005** — Migration that creates these tables in the database.
